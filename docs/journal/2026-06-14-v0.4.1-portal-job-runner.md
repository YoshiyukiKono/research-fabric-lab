# v0.4.1 Portal Job Runner

Date: 2026-06-14

## Summary

Research Fabric Lab において、Research Portal から Kubernetes Job を起動し、実行結果を MinIO に保存する機能を実装した。

これにより、

Research Goal
↓
OpenAI
↓
Experiment Spec
↓
Portal
↓
Kubernetes Job
↓
Artifact
↓
MinIO

という End-to-End の研究実行フローが成立した。

---

## Background

v0.4.0 では Kubernetes Job を直接適用することで実験実行を行っていた。

```bash
kubectl apply -f deploy/k8s/jobs/ising-1d-demo/job.yaml
```

これは実験実行基盤としては機能するものの、

* Portal から実行できない
* ユーザー操作が kubectl に依存する

という課題があった。

今回の目的は、

```text
Portal
↓
Run Button
↓
Job Creation
```

を実現することである。

## Step 1: RBAC

Research Portal Pod から Kubernetes Job を作成できるようにするため、ServiceAccount を導入した。

追加ファイル:

```text
serviceaccount.yaml
role.yaml
rolebinding.yaml
```

権限:

```text
batch/jobs
create
get
list
watch
delete
```

Pod / Log 参照権限も付与した。

---

## Step 2: ServiceAccount Assignment

Deployment に以下を追加した。

```yaml
serviceAccountName: research-portal
```

確認:

```bash
kubectl get deploy research-portal \
  -n research \
  -o yaml
```

---

## Step 3: Kubernetes Python SDK

Portal から Kubernetes API を呼び出すため、

```text
kubernetes
```

パッケージを追加した。

Deployment:

```bash
pip install streamlit openai kubernetes
```

確認:

```bash
kubectl exec -n research deploy/research-portal \
  -- python -c "import kubernetes; print(kubernetes.__version__)"
```

結果:

```text
36.0.2
```

---

## Step 4: Run Button

Portal に

```text
Run ising-1d-demo Job
```

ボタンを追加した。

実装内容:

* in-cluster config を利用
* Kubernetes API を利用
* Job を動的生成
* namespace = research

---

## Step 5: Artifact Versioning

当初の Job は

```text
ising-1d-demo/result.json
```

へ保存していた。

その場合、

```text
毎回上書き
```

となる。

実験履歴を残すため、

```text
run_id
```

を導入した。

生成例:

```text
20260614-073546-9e6037
```

保存先:

```text
research-artifacts/
└── ising-1d-demo/
    └── runs/
        └── 20260614-073546-9e6037/
            └── result.json
```

## Troubleshooting

### Kubernetes SDK Installation

Portal 再起動後、

ログに Kubernetes SDK が見えないように見えた。

確認:

```bash
kubectl exec -n research deploy/research-portal \
  -- python -c "import kubernetes"
```

結果:

```text
success
```

実際には正常にインストールされていた。

---

### Deployment Rollout

ConfigMap 更新後は、

```bash
kubectl rollout restart deploy/research-portal \
  -n research
```

を実施した。

その後:

```bash
kubectl rollout status deploy/research-portal \
  -n research
```

で正常ロールアウトを確認した。

## Lessons Learned

### Research Portal は Kubernetes Client になれる

Portal は単なる UI ではなく、

```text
Portal
↓
Kubernetes API
↓
Job
```

の制御プレーンとして機能できる。

---

### Artifact は履歴管理が重要

結果ファイルを固定パスへ保存すると、

実験履歴が消失する。

run_id を導入することで、

研究実行履歴として扱えるようになった。

---

### Workload Cluster と Management Cluster の分離

Portal は Workload Cluster に存在する。

Artifact Storage は Management Cluster に存在する。

この構成により、

将来的な

```text
agent-lab
gpu-lab
quantum-lab
```

への拡張が容易になる。
